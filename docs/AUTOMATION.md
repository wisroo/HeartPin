# HeartPin Automation Plan

작성일: 2026-06-19

이 문서는 HeartPin을 두 대 이상의 컴퓨터와 Codex Cloud/Local 환경에서 일관되게 개발하기 위한 자동화 기준을 정의한다.

## 목표

- Cloud에서는 GitHub에 올라간 코드만으로 빌드와 정적 점검이 재현되어야 한다.
- Local에서는 사진, 브라우저 권한, EXIF, 외장하드처럼 실제 기기 상태가 필요한 흐름을 직접 검증한다.
- 두 컴퓨터는 같은 저장소와 같은 명령을 사용하되, 같은 스케줄 자동화를 동시에 켜서 중복 실행하지 않는다.
- 자동화는 초기에는 보고 전용으로 운영하고, 파일 수정 자동화는 결과 품질을 확인한 뒤 별도로 허용한다.

## 환경 기준

권장 로컬 기준:

- Node.js: `20.20.2`
- npm: `10.8.2`
- 의존성 설치: `npm ci`
- 빌드 검증: `npm run build`

다른 컴퓨터를 준비할 때는 저장소를 클론한 뒤 다음 순서로 확인한다.

```bash
npm ci
npm run build
```

나중에 `.env.local`이 필요한 기능이 추가되면 `.env.example`에 필요한 키를 문서화하고, 실제 값은 각 컴퓨터에 따로 둔다.

## Cloud-Friendly Automation

Codex Cloud 또는 GitHub 기반 환경에서 돌릴 수 있는 자동화다. 로컬 파일, 외장하드, 개인 사진 폴더, 브라우저 권한에 의존하지 않아야 한다.

### Weekly Build And Docs Check

권장 주기: 주 1회

권장 작업:

1. `npm ci`를 실행한다.
2. `npm run build`를 실행한다.
3. `README.md`, `docs/ROADMAP.md`, 실제 `src/` 구조 사이의 괴리를 확인한다.
4. 발견사항만 보고한다.
5. 파일을 자동 수정하지 않는다.

권장 프롬프트:

```text
HeartPin 저장소를 주간 점검해줘.

- npm ci 후 npm run build를 실행한다.
- 빌드 실패 시 원인과 의심 파일을 요약한다.
- README.md, docs/ROADMAP.md, src/ 구조를 비교해서 문서가 오래되었거나 실제 코드와 어긋난 부분을 보고한다.
- 이 자동화는 보고 전용이다. 파일을 수정하거나 커밋하지 말고, 필요한 후속 작업만 제안한다.
- 사진 원본, 외장하드, 브라우저 권한, 로컬 .env 값이 필요한 검증은 Local-only 항목으로 분리해서 표시한다.
```

## Local-Only Checks

아래 항목은 실제 컴퓨터와 브라우저 상태가 필요하므로 Codex Cloud에서 검증하지 않는다.

- 사진 업로드 UX
- JPEG EXIF 촬영일/GPS 파싱
- HEIC 또는 모바일 사진 선택기 동작
- File System Access API
- 외장하드 연결, 스캔, 복사, 삭제 후보 흐름
- 브라우저 권한 프롬프트
- 실제 지도/타일 렌더링 확인

권장 로컬 수동 점검:

```bash
npm ci
npm run dev
```

브라우저에서 다음 흐름을 직접 확인한다.

1. 사진 업로드
2. 위치 있는 사진과 위치 없는 사진 분류
3. 새 여행 미리보기
4. 지도 핀/갤러리/정리함 전환
5. 외장하드 또는 로컬 폴더 연동 기능이 추가된 뒤에는 실제 파일 쓰기/읽기 권한

## Two-Computer Operating Rule

두 컴퓨터 모두 같은 방식으로 개발할 수 있게 준비하되, 같은 정기 자동화는 기본적으로 한 컴퓨터 또는 Cloud 한 곳에서만 활성화한다.

권장 운영:

- Cloud 또는 주 컴퓨터: 주간 build/docs 보고 자동화 활성화
- 보조 컴퓨터: 같은 명령으로 수동 검증 가능하게 유지
- Local-only 검증: 실제 사진/외장하드가 연결된 컴퓨터에서만 실행

중복 실행을 피해야 하는 이유:

- 같은 보고가 두 번 생성된다.
- worktree와 캐시가 불필요하게 늘어난다.
- 자동 수정 권한을 나중에 켤 경우 충돌 가능성이 커진다.

## Worktree Guidance

Git 프로젝트에서 Codex Local 자동화는 가능하면 worktree에서 실행한다.

worktree를 쓰는 이유:

- 현재 작업 중인 로컬 파일을 건드리지 않는다.
- 자동화 결과를 독립적으로 검토할 수 있다.
- 나중에 필요한 경우 branch/PR로 승격하기 쉽다.

단, 외장하드, `.env.local`, 로컬 사진 샘플처럼 git에 없는 파일이 필요한 검증은 worktree에 자동으로 존재하지 않는다. 그런 파일이 꼭 필요해지면 `.worktreeinclude`에 복사 대상 패턴을 명시한다.

## 변경 자동화 도입 조건

보고 전용 자동화가 안정적으로 동작한 뒤에만 파일 수정 자동화를 추가한다.

수정 자동화는 다음 조건을 만족해야 한다.

- 수정 범위가 명확하다.
- 빌드 또는 테스트 명령이 있다.
- 결과 diff가 작고 리뷰 가능하다.
- 커밋/PR 생성 여부를 명확히 정한다.

현재 단계에서는 자동 수정보다 보고 전용 자동화를 우선한다.
