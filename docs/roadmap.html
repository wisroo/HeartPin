1 · SERVICE

서비스 정의와 핵심 가치

갤러리에 흩어진 사진을 예쁘게 정리하며 기록하는 커플 전용 웹 — Trip → Day → Spot → Moment

맡는 것 ✅

  • 기록으로서의 제공 — 압축본 + 설명 + 여정 트리의 영구 보존
  • 서로 간 전송의 용이성 — 클라우드 경유 중계(우체통)
  • 소재 추적 — 어떤 사진이 어디(폰·PC·하드)에 있는지 아는 장부

맡지 않는 것 ❌

  • 원본 백업 — 원본은 각자 로컬/외장하드에서 관리
  • 원본의 영구 클라우드 보관
  • 타인 사진의 운영(SaaS) — 프라이버시상 배제
📒
핵심 개념 — 원장(臺帳, Ledger) 모델

원본이 어디에 흩어져 있든, "무엇이 어디에 있고 어디로 가야 하는지"를 아는 유일한 장부가 된다. 기록은 공동, 원본은 소유자 — 콘텐츠 해시(SHA-256)가 원본↔압축본↔기록을 잇는다. 동기화 규칙은 단 하나: "외장하드(정본 보관소)가 전체 집합을 갖게 하라."

2 · DECISIONS

핵심 아키텍처 결정

결정 + 근거 + 배제한 대안 (ADR)

D1

기록은 클라우드 필수

둘이 어디서나 봐야 함 — 로컬은 남이 못 보고, 외장하드는 꽂혀 있을 때만 존재

배제: 로컬 전용(IndexedDB) — 기기 1대에 갇힘
D2

클라우드는 "각 커플의 것" (BYO)

운영자가 타인의 사진을 관리하지 않음 — 개인정보 책임·비용·유출 리스크 구조적 제거

배제: 운영형 SaaS
D3

1차 백엔드 = Supabase (서울)

DB+Storage+Auth 올인원, 서버 코드 최소, 오픈소스(락인 없음), 무료 티어

배제: FastAPI 상시 서버 Firebase PocketBase
D4

StorageAdapter 추상화 + 내보내기

백엔드 교체 가능(추후 Google Drive 어댑터) · "내 데이터는 언제든 들고 나간다"

배제: Supabase 직결 하드코딩
D5

클라우드엔 압축본만

display 2000px + thumb 400px — 12MB → ~330KB (97%↓), 무료 1GB로 ~3,000장

배제: 원본 업로드 — 비용·"백업 아님" 정체성 충돌
D6

정본 보관소 = 공동 외장하드 1곳

위치가 N개여도 규칙이 하나로 수렴 — 복잡도 폭발 방지

배제: NAS(사용자 결정) 전 위치 대등 동기화
D7

공유 계정 1개 + 캐릭터 owner

2인 전용 — 가입/초대/권한 불필요. 업로드 시 바라/뇽이 토글로 소유자 기록

배제: 계정 2개+권한 모델 — 과함
D8

보관함 연결 = FS Access API

웹앱이 로컬 파일을 읽고·쓰고·지울 유일한 표준. 핸들 영속화로 재연결 1클릭

제약: 데스크톱 크롬/엣지 전용
D9

전송 = 클라우드 경유 "우체통"

원본 임시 중계 → 수령 후 삭제. 폰↔PC가 시간차를 두고 만나도 됨

배제: P2P(WebRTC) — 동시 접속 필요, 복잡도 과다
D10

모바일: 웹+PWA → Capacitor 앱

적응형 셸을 한 코드베이스에서 — 같은 코드 ~90% 재사용으로 스토어 앱까지

배제: React Native 재작성 — +1~2달
D11

서버 코드는 Vercel Functions 최소

Supabase가 API 대체. LLM/Mapbox 키 은닉 등 필요한 함수만 (JS·Python 혼용 가능)

FastAPI는 Python 친숙성 사안일 뿐 — 필수 아님
D12

Phase 0 데모 = 로컬 Express 서버 (임시)

6시간 시연: 폰 2대 업로드→지도 기록→하드 저장. 하드가 서버에 직결 → Node fs 직접 쓰기로 FS API·우체통 없이 "폰→하드"가 한 홉. record.json 필드 = Supabase 스키마(전환 보장)

배제: Supabase 압축 일정(6시간 내 불가) FS Access API(폰 미지원)

0

데모 — "로컬 시연"

~6시간🎬 지금 여기 (06-06)

데모 정의 — 두 사람이 폰으로 로컬 서비스에 접속해 사진을 올리면 지도에 기록되고, 원본·압축본·기록이 연결된 외장하드에 저장된다. 시연 + MVP 테스트 + 사진 정리의 시작.

재편 배경: 폰에 분산된 사진을 하드로 옮길 시간 부족 + 시연 우선 → 폰 직접 업로드 경로(원 Phase 2~3 일부)를 로컬 서버 방식으로 앞당김 (D12)

0-1로컬 서버 — Express·multer·sharp·exifr: 업로드→EXIF·해시(서버 계산)→WebP 압축→하드 저장·record.json~1h
0-2데이터 레이어 교체 — 메모리 → src/api.js(어댑터 시임) + 3초 폴링 · 샘플 제거(빈 상태 시작)~1.5h
0-3a업로드 배관 — 서버 업로드 · 가짜 메타 폴백 제거→'위치 없음'. 여기까지만 돼도 데모 성립~0.5h
0-3b모바일 업로드 플로우(디자인 #4) — 런치(owner 1회 선택·D+N 자동, 2024-06-29=D+1)→고르기(OS 선택기)→읽는중(실진행률)→한 장씩 확인(세션 부트스트랩: 자동 이름→수정→근접 제안)→완료. 건너뛴 사진은 정리함~2h
0-4실기기 테스트 — 아이폰 EXIF/HEIC · AP isolation · 방화벽 (0-3a 후 1차, 0-3b 후 2차)~1h
0-5하드 폴더 검증 + 리허설 (압축)~0.5h
완료 기준 (DoD)
  • 폰 2대 업로드 → 정리함 배치 → Mac 지도에 기록 표시
  • 하드에 원본+압축본+record.json 생성, 서버 재시작에도 유지
  • 해시 중복 감지 + owner(바라/뇽이) 기록
  • EXIF 없는 사진은 '위치 없음' 탭으로 (가짜 좌표 폴백 제거)
  • 모바일: 런치→고르기→한 장씩 확인→완료 플로우로 업로드·배치 (디자인 #4)

MVP — "기록의 시작" (Supabase 전환)

~4.5–5.5일🎯 MVP

MVP 정의(확정) — 외장하드 연결로 웹에서 데이터를 업로드해서 기록하고, 상대방도 접속 가능하게 하여 양쪽 모두 업로드할 수 있도록 한다.

Phase 0 산출물 재사용(api.js 시임 · 업로드 로직 · 하드 폴더/record.json 포맷)으로 원 5–6일에서 소폭 단축

완료 기준 (DoD)
  • 두 사람이 각자 PC에서 로그인 → 업로드 → 기록 생성, 다른 기기에서도 유지
  • 업로드 시 owner 기록 + 동일 파일 중복 감지
  • 하드 연결 → 새 사진 복사 + 흐릿 삭제 실행(설명·기록은 보존) + 백업 JSON
  • 하드 분리 상태에서도 정상 동작, 할 일은 큐에 적재

의도적으로 뺀 것: 우체통(원본 릴레이) · 모바일 최적화(기본 반응형만) · 보관 현황 대시보드 · LLM/역지오코딩

7 · DISCUSSION LOG

논의 기록 요약

결정에 이른 경로 — 추후 "왜 이렇게 했지?"의 답

  1. 백엔드 스택 — FastAPI 필요? → Supabase로 서버 코드 최소화. Node=JavaScript(Java 아님). 필요 함수만 Vercel Functions(JS·Python 혼용)
  2. 프론트만 영속화? → localStorage/IndexedDB는 "그 기기 그 브라우저"에 갇힘 → 클라우드 필요
  3. 사진 용량 → 원본 안 올림. 압축본만(97% 절감). 원본 보관은 폰/iCloud/하드의 몫
  4. "2명 업로드"는 공수 변수 아님 — 공유 계정이면 코드 동일. 진짜 분기는 "클라우드 vs 로컬"
  5. 공동관리 → 원장 모델: 해시 + owner + original_status. 흐릿 삭제는 "삭제 예약 큐"로 비동기 처리
  6. 외장하드 vs 각자 관리 → 두 옵션이 아니라 "보관함 연결" 단일 메커니즘 — 어떤 폴더를 연결하느냐의 차이
  7. 모바일→하드 전송 → 클라우드 릴레이(우체통)로 가능. 동시 사용 가능 — 단 역할 고정(정본=하드 1곳)
  8. 운영 책임 → 타인 사진 관리 거부 → BYO-cloud 템플릿 배포 모델 확정
  9. Supabase 필수? → 아니다, "공수 최소 선택". 어댑터+내보내기로 락인 차단. 비개발자의 답은 Drive 어댑터
  10. 모바일 → 적응형 웹+PWA 먼저, 스토어 앱은 Capacitor 래핑. 하단 탭바·바텀시트를 한 코드베이스 조건부 렌더링으로 — 지금 한꺼번에 개발 가능
  11. 시연 우선 재편 (06-06) → 6시간 뒤 시연 + 폰 사진을 하드로 옮길 시간 부족 → Phase 0(로컬 Express 서버) 신설. 하드가 서버에 직결이라 폰 업로드가 곧 "폰→하드 수송" — 우체통·FS API 없이 데모 충족 (D12)
  12. Supabase 전환 보장 장치 → record.json 필드 = Supabase 스키마, src/api.js 단일 시임. 전환 = 어댑터 구현 교체 + 일회성 마이그레이션. 데모는 빈 상태에서 실데이터로 시작, EXIF 없으면 '위치 없음' 플로우(가짜 좌표 폴백 제거)
  13. 모바일 업로드 와이어프레임 #4 확정 (06-06) → 런치→고르기→읽는중→한 장씩 확인→완료. 웹 번역 2건: 카메라롤 그리드=OS 선택기 경유, 장소명=세션 부트스트랩(자동 이름→수정→근접 300m 제안, 역지오코딩 오면 자동화). owner는 런치 첫 1회 선택, D+N은 기념일 설정값(2024-06-29=D+1) 자동 계산. 모바일은 업로드→즉시 배치, 건너뛴 사진만 정리함
