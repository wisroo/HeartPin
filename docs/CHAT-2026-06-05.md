## Q1. FastAPI가 따로 필요한가?

**결론: 필수는 아니고, "Python 생태계가 필요한가"로 결정하면 됩니다.**

| | Node 함수 (`api/*.js`) | FastAPI |
|---|---|---|
| 인프라 | Vercel에 그대로 (추가 0) | Vercel Python 런타임으로 같은 레포 가능¹, 또는 별도 서버(Cloud Run 등) |
| HEIC 처리 | `exifr`(메타 추출 OK), 변환은 wasm 기반이라 다소 느림 | **`pillow-heif`로 메타+변환 모두 깔끔** — 사실상 Python의 최대 강점 |
| LLM 대사·역지오코딩 프록시 | fetch 한 번이라 충분 | 동일하게 가능 (과함) |
| 향후 ML(흐릿한 사진 판별 등) | 약함 | 강함 |

¹ Vercel은 `api/` 폴더에 Python 함수도 지원하고, FastAPI 자체도 서버리스로 올라갑니다. 단 서버리스 제약(콜드 스타트, 실행시간 제한, 번들 250MB)은 동일하게 받아요. JS 함수와 Python 함수를 **같은 `api/`에 섞는 것도 가능**합니다.

**추천 경로:** 지금 목록(역지오코딩 프록시, LLM 대사, 영속화 API)은 Node 함수로 시작 → **HEIC 변환/ML이 실제로 필요해지는 시점에 그 엔드포인트만 Python으로** 추가. 처음부터 FastAPI 서버를 따로 세우는 건 2인용 서비스엔 운영 부담만 늘립니다. 다만 본인이 Python이 훨씬 편하다면 처음부터 `api/`를 전부 Python(FastAPI)으로 가도 무방해요 — 구조는 동일하게 한 레포입니다.

## Q2. 업로드 정보 영속화는 어떻게?

현재는 사진이 `URL.createObjectURL()`(메모리 blob URL)이고 데이터 트리도 메모리 상태라 **새로고침하면 전부 사라집니다.** 영속화는 두 덩어리로 나뉩니다:

```
[사진 원본 = 바이너리]              [구조 데이터 = Trip 트리/정리함/대사]
        ↓                                    ↓
   Object Storage                        Database
 (Vercel Blob / Supabase Storage)   (Postgres: Neon / Supabase)
```

**업로드 흐름이 이렇게 바뀝니다:**

1. 클라이언트: 파일 선택 → JPEG면 EXIF를 지금처럼 브라우저에서 읽음 (HEIC면 `api/exif`로 보내 메타 추출)
2. 원본을 Storage에 업로드 → **영구 URL** 획득
3. `inbox_items` 테이블에 insert (영구 URL + 촬영일시 + GPS + 라벨)
4. 정리함에서 배치/새 여행 → `trips/days/spots/moments` 테이블에 기록
5. 앱 시작 시 DB에서 트리를 fetch → 지금의 `HP_DATA` 메모리 변이 자리를 API 호출로 교체

**테이블은 현재 데이터 구조 그대로 매핑됩니다** (이미 코드가 트리 구조라 스키마 설계가 끝나 있는 셈):

```sql
trips(id, region, title, start_date, date_label, tags)
days(id, trip_id, label, date)
spots(id, day_id, name, time, lat, lng, mood, guide, reaction, sort_order)
moments(id, spot_id, url, label, ratio, tint)
inbox_items(id, kind, date, time, lat, lng, url, label, blur)
```

**스택 선택지 3단계:**

| 수준 | 구성 | 특징 |
|---|---|---|
| **0. 서버 없이** | localStorage(트리) + IndexedDB(사진 blob) | 30분 작업, 배포 전 임시용. **단, 기기 1대에 갇힘** — 커플 둘이 못 봄 |
| **1. Supabase (추천)** | Postgres + Storage + Auth 올인원 | 커플 2명 로그인(Auth)까지 한 번에. 무료 티어로 충분. 프론트에서 직접 호출 가능해서 `api/` 함수도 최소화됨 |
| **2. Vercel 순정** | Vercel Blob + Neon Postgres | Vercel 콘솔 안에서 끝나는 게 장점. Auth는 따로 구현 필요 |

"커플 전용 비공개"라는 PRD 요구(둘 다 자기 폰/PC에서 접근 + 로그인)까지 생각하면 **Supabase가 가장 적은 코드로 전부 해결**됩니다. 이 경우 FastAPI/Node 함수의 역할은 더 줄어서, 비밀키가 필요한 것(LLM, Mapbox 토큰)과 HEIC 처리만 `api/`에 남기면 돼요.

## 🤖 Claude

네, 맞습니다 — 앞서 말한 storage(Supabase Storage, Vercel Blob)와 DB(Neon, Supabase Postgres)는 **전부 클라우드 관리형 서비스**예요. 사진 원본이 물리적으로 그쪽 클라우드 서버에 저장됩니다.

**왜 클라우드가 전제냐면:**

1. **Vercel에는 "서버 디스크"가 없습니다.** Vercel 함수는 서버리스라 실행이 끝나면 사라지고, 파일을 저장할 영구 디스크 자체가 없어요. 전통적인 VPS처럼 "서버에 파일 저장"이 불가능해서, 바이너리는 반드시 외부 storage로 나가야 합니다.
2. **커플 2명이 각자 기기에서 같은 기록을 봐야** 하니까요. 기기 안에만 저장(localStorage/IndexedDB)하면 내 폰에 올린 사진을 상대방이 못 봅니다.

**민감한 부분(커플 사진)이라 짚어두면:**

- 버킷을 **private**으로 두고 서명된 URL(signed URL) 또는 RLS(행 단위 권한)로 **로그인한 두 사람만** 접근하게 설정합니다. 기본 비공개라는 PRD 요구와 맞아요.
- Supabase는 리전 선택이 가능해서 **서울(AWS ap-northeast-2)** 에 둘 수 있습니다.
- 비용: 무료 티어(Supabase 기준 storage 1GB)로 시작 가능. 여행 사진이 쌓이면 1GB는 금방이라, 장기적으론 유료 플랜(월 $25, 100GB) 가능성은 염두에 두세요.

**클라우드가 영 꺼려진다면** 대안은 자가 호스팅(집 NAS나 미니PC에 Supabase self-hosted / MinIO + DB)인데, 외부 접속 설정·백업·보안 관리를 직접 해야 해서 "일회성 배포" 취지와는 안 맞습니다. 현실적으로는 **private 버킷 + 서울 리전 클라우드**가 균형점이에요.

정리하면: 구조 데이터든 사진이든 "우리 둘이 어디서나 보는" 서비스가 되는 순간 클라우드 저장이 사실상 필수이고, 접근 제어로 비공개를 보장하는 방식입니다.

## Q. 프론트만으로 영속화하면 데이터가 어디에 저장되나?

**그 기기, 그 브라우저의 프로필 폴더 안 디스크**에 저장됩니다.

| 저장소 | 실제 위치 (맥 크롬 예) | 용량 | 특징 |
|---|---|---|---|
| localStorage | `~/Library/Application Support/Google Chrome/Default/Local Storage/` | ~5MB | 문자열만 — 기록 트리 정도 |
| IndexedDB | 같은 프로필 폴더 내 `IndexedDB/` | 수 GB | 사진 blob도 저장 가능 |

제약이 명확해요: ① 같은 PC라도 **다른 브라우저면 안 보임**, ② 다른 기기(폰)에서 안 보임, ③ 브라우저 "사이트 데이터 삭제"하면 **기록이 통째로 날아감**. 그래서 프론트만 영속화로 가면 기록 JSON을 사진 폴더에 자동 백업하는 장치가 필수입니다.

## "원본은 로컬, 흐릿한 사진은 로컬에서도 삭제" — 기술적 진실

일반 웹앱은 사용자 디스크의 파일을 못 지웁니다. 유일한 방법은 **File System Access API**: 사용자가 사진 폴더를 한 번 선택해 권한을 주면, 앱이 그 폴더를 직접 읽고(업로드 없이 표시 가능) **파일 삭제까지** 할 수 있어요. 단 **크롬/엣지 데스크톱 전용** (아이폰 사파리 불가) → "정리 작업은 PC에서, 모바일은 열람용"이라는 제약이 생깁니다. 이 요구사항은 어떤 모드를 골라도 이 API로 구현하게 됩니다.

## Q. 백엔드 공수는 실제로 얼마나?

중요: Supabase를 쓰면 **서버 코드를 거의 안 짭니다** — DB·스토리지·인증·API를 Supabase가 대신하고, 프론트가 SDK로 직접 호출해요. "백엔드 개발"의 실체는 서버 구축이 아니라 **프론트의 데이터 레이어 교체** 작업입니다.

**Mode A — 완전 로컬 (백엔드 0, 혼자·PC 1대)** : 합계 **~3일**
| 작업 | 공수 |
|---|---|
| 데이터 레이어 추상화 + IndexedDB 영속화 | ~1일 |
| 사진 폴더 연결(FS API) + 폴더에서 직접 표시 | ~1일 |
| 흐릿 "버리기" → 로컬 삭제 (설명 기록은 보존) | ~0.5일 |
| 기록 JSON 사진 폴더 자동 백업 | ~0.5일 |

**Mode B — 클라우드 기록 (Supabase, 둘이 올리고 둘이 봄)** : 합계 **~4–5일**
| 작업 | 공수 |
|---|---|
| Supabase 셋업 + 테이블 5개 + 공유 계정 인증 | ~1일 |
| 업로드 파이프라인 (클라 압축 → Storage → DB) | ~1일 |
| 데이터 레이어 교체 (HP_DATA → 조회/변이) | ~1.5–2일 |
| 로그인 화면 + Vercel 배포 연결 | ~0.5일 |
| (옵션) FS API 로컬 삭제 연동 — A의 그 기능 | +0.5–1일 |

차이가 **하루 이틀**입니다. 그리고 A에서 B로 나중에 옮기려면 데이터 레이어를 또 갈아끼워야 해서, "둘이 본다"가 서비스의 핵심 가치라면 처음부터 B가 총공수가 더 적어요.

## Q. FastAPI / Node / 자바?

한 가지 정정: **Node는 자바(Java)가 아니라 자바스크립트(JavaScript)** 입니다 — 지금 프론트와 같은 언어라서 레포 전체가 한 언어로 통일돼요. 그리고 위 구성(Supabase)에서는 직접 짤 서버 코드가 거의 없어서 **언어 선택의 무게 자체가 작습니다**. 나중에 LLM 대사 생성이나 HEIC 변환 함수가 필요해지면, 그 함수 1~2개만 Python으로 짜도 됩니다(Vercel은 `api/`에 JS·Python 혼용 가능) — FastAPI 서버를 따로 세울 일은 없어요.

좋은 방향이에요. 이 구조에서 "공동관리"의 본질은 **원본 파일을 한곳에 모으는 게 아니라, 흩어진 원본들을 공유 기록(DB)이 추적·지휘하게 만드는 것**입니다. 정리하면:

## 핵심 모델: "기록은 공동, 원본은 소유자" + 해시로 연결

원본은 각자 기기에 있으니, 클라우드 기록이 **원본의 신분증명서** 역할을 하면 됩니다. 업로드할 때 파일의 **콘텐츠 해시(SHA-256)** 를 계산해서 메타데이터와 함께 저장하는 게 열쇠예요:

```
moments / inbox_items에 추가될 필드
├── content_hash      ← 원본↔압축본↔기록을 잇는 열쇠 (파일명 바뀌어도 추적)
├── original_name     IMG_1234.HEIC
├── owner             'bara' | 'nyong'  ← 누구의 원본인지
└── original_status   'kept' | 'discard_pending' | 'discarded'
```

`owner`는 공유 계정이라도 **업로드할 때 "바라로/뇽이로" 캐릭터 토글**로 받으면 됩니다 — 서비스 컨셉(두 주인공)과 자연스럽게 맞아요.

이 모델로 공동관리 기능들이 이렇게 풀립니다:

**① 흐릿 사진 삭제 — "삭제 예약 큐" (핵심)**
내가 정리함에서 상대방이 올린 사진을 "버리기" 해도, 그 원본은 상대방 기기에 있어서 즉시 못 지웁니다. 대신:
- 클라우드 기록에서는 바로 처리 (압축본 제거, **설명·기록 행은 보존** — 말씀하신 요구사항)
- DB에 `discard_pending` 마킹 → **원본 소유자가 다음에 PC에서 사진 폴더를 연결하면** "지난번에 버리기로 한 23장, 로컬에서도 지울까요?" 일괄 제안 → 해시로 파일 매칭해서 삭제

즉 둘이 비동기로 정리해도 결국 양쪽 로컬까지 정리되는 **최종 일관성** 모델입니다.

**② 중복 감지**
커플은 같은 장면을 둘 다 찍거나 카톡으로 주고받아서 중복이 많죠. 업로드 시 해시 대조로 "이 사진, 뇽이가 이미 올렸어요" 처리가 공짜로 생깁니다 (완전 동일 파일 기준 — 비슷한 사진 감지는 추후 perceptual hash로 확장 가능).

**③ 원본 찾기**
기록에서 사진을 보다가 "원본 필요"(인화, 보정 등) → 각 사진에 "원본: 바라 PC" 뱃지 → 그 사람이 폴더 연결하면 해시·파일명으로 스캔해 원본을 찾아 열어줌. 원본이 사라졌으면 `lost` 표시 — 그래도 기록(압축본+설명)은 살아있음.

## 그럼 원본 자체의 관리는 어떻게?

서비스가 원본 백업을 안 하는 대신, 권장 운영을 단계로:

| 단계 | 방법 | 비고 |
|---|---|---|
| **기본** | 각자 폰 자동백업(iCloud/구글포토) 유지 + 서비스의 압축본을 "준백업"으로 | 압축본을 긴 변 2000px 정도로 넉넉히 잡으면, 원본을 잃어도 기록은 화면용으로 충분한 품질로 생존 |
| **추천** | **공동 외장하드/NAS 1개를 "원본 금고"로 지정** — 여행 끝나면 둘 다 거기로 원본을 모으고, HeartPin 폴더 연결도 그 디스크로 | 물리적 공동관리. 삭제 예약 큐도 한 곳에서 실행되니 가장 깔끔. 실제 커플들 워크플로(여행 후 PC에서 몰아서 정리)와 일치 |
| **확장(추후)** | "이 사진은 원본도 보관" ⭐ 선택 기능 — 찜한 사진만 R2($0.015/GB/월)에 원본 업로드 | 전체 원본 백업의 비용 부담 없이 소중한 컷만 이중 보관 |

**솔직한 제약 한 가지:** 원본 삭제·폴더 스캔(FS Access API)은 **데스크톱 크롬/엣지에서만** 됩니다. 그래서 자연스러운 역할 분담은 — **모바일 = 열람 + 가벼운 업로드, PC = 원본 정리 세션**. 이건 한계라기보다 "여행 다녀와서 PC 앞에 앉아 같이 정리하는 의식"이라는 서비스 경험으로 만들 수 있는 부분이에요.

## 공수 영향

이전 Mode B(~4–5일) 위에 해시 계산 + owner 토글 + 삭제 예약 큐 + 폴더 매칭으로 **+1.5~2일** 정도입니다. 단계를 나누면: 해시·owner·중복감지는 처음부터 넣는 게 좋고(나중에 넣으면 기존 사진 해시가 없음), 삭제 예약 큐·원본 찾기는 2차로 미뤄도 됩니다.

시작한다면 순서는 ① Supabase 스키마(위 필드 포함) → ② 업로드 파이프라인(압축+해시+owner) → ③ 데이터 레이어 교체 → ④ 정리 큐/폴더 연결 순서를 추천해요. ①②부터 잡아볼까요?

## 1. 각자 클라우드/디바이스 간 전송 — 답: 서비스 클라우드를 "경유지"로

기기끼리 직접 통신할 필요가 없습니다. **모든 사진이 어차피 업로드 때 서비스 클라우드를 한 번 지나가요.** 이 길목을 전송 채널로 쓰면 됩니다:

```
바라 기기 ──원본──→ [서비스 클라우드] ──→ 뇽이 기기
                    ├ 압축본: 영구 보관 (기록용)
                    └ 원본: "우체통" — 전송용 임시 보관, 수신 확인되면 삭제
```

**3단계로 나누면:**

| 단계 | 방식 | 공수 |
|---|---|---|
| ① 압축본 가져오기 | 이미 클라우드에 있음 → "내게 없는 사진" 필터에서 바로 다운로드 | 거의 0 |
| ② 원본 우체통 (추천) | B가 "원본 요청" → A가 다음 접속 때 원본 업로드 → B 다운로드 → **클라우드에서 삭제** (또는 7일 후 자동 삭제) | +1~1.5일 |
| ③ P2P(WebRTC) 직접 전송 | 둘 다 동시 접속 필요, 네트워크 이슈 많음 | 비추천 — 복잡도 대비 가치 낮음 |

②가 중요한 이유: **"백업은 안 하지만 전송은 맡는다"는 목표와 정확히 일치**합니다. 원본이 클라우드에 머무는 건 전송되는 동안뿐이라 저장 비용이 안 쌓이고, 서비스의 정체성(기록 서비스, 백업 아님)도 유지돼요. 한 가지 주의: 원본 일괄 전송 시 용량이 잠깐 튀므로(200장×8MB≈1.6GB) 배치 단위 제한을 두거나, 우체통 버퍼만 Cloudflare R2(무료 10GB)로 분리하면 됩니다.

**"내 개인 클라우드로 가져오기"의 현실:** iCloud는 외부 API가 없고 Google Photos API도 제약이 많아 **직접 연동은 비현실적**입니다. 현실적인 경로는 — PC에서는 FS Access API로 **내 사진 폴더에 직접 쓰기**(그 폴더가 iCloud Drive/구글드라이브 동기화 폴더면 자동으로 개인 클라우드행), 모바일에서는 카메라롤 저장 → 자동 백업. 즉 "개인 클라우드 연동"이 아니라 "동기화 폴더에 넣어주기"로 풀어야 해요.

**"나에게 없는 것만" 추적은** 이전에 설계한 `owner` + `content_hash`로 충분합니다. 사진마다 `copies: {bara: 'original', nyong: 'none'|'fetched'}` 상태를 두고, 폴더 연결 시 해시 스캔으로 실제 보유 여부를 갱신 → "뇽이에게 없는 사진 12장" 필터가 나옵니다.

## 2. 공동 외장하드 UX

FS Access API의 디렉터리 핸들은 **영속화가 됩니다** — 처음 한 번 폴더를 선택하면, 다음부터는 "외장하드 다시 연결" 버튼 한 번(권한 확인 클릭 1회)으로 복구돼요. 설계 제안:

- **연결을 일급 개념으로**: 상단에 `🔌 외장하드: 연결됨/분리됨` 뱃지. 분리 상태여도 서비스는 정상 동작하고, 하드가 필요한 작업은 큐에 쌓임
- **연결되는 순간 큐 일괄 실행**: "외장하드가 연결됐어요 — 할 일 3건: 흐릿 사진 14장 삭제 / 새 사진 31장 하드로 복사 / 기록 백업 갱신" → 원클릭 실행. 바라가 안내하는 위저드로 만들면 서비스 톤과 맞아요
- **폴더 컨벤션**: `HeartPin/원본/2025-10-부산/…` + `HeartPin/record-backup.json` (기록 자체의 백업도 하드에 — 클라우드가 사라져도 기록 생존)
- 부수 효과: **외장하드를 서로 주고받는 것 자체가 전송 수단**이 됩니다. 서비스는 "하드에 뭐가 있는지"를 알고 있으니, 누가 꽂든 그 시점에 정합성을 맞춰주는 역할만 하면 돼요

## 3. 서비스 가치 확장 프레임: "커플 사진의 원장(臺帳)"

지금까지 논의를 하나로 묶으면, HeartPin의 확장 가치는 이겁니다:

> 원본이 어디에 흩어져 있든(바라 폰, 뇽이 PC, 외장하드), **"무엇이 어디에 있고, 어디로 가야 하는지"를 아는 유일한 장부**가 된다. 파일 저장은 안 맡되, 소재 추적과 이동 지시를 맡는다.

이 프레임에서 나오는 기능들 — 우선순위 순으로:

1. **보관 현황** (장부의 얼굴): "우리 기록 1,240장 — 바라 800 · 뇽이 900 · 외장하드 1,100 · ⚠️ 원본이 어디에도 없는 사진 3" → 정리 동기 부여 + "하드에 모으기" 자연 유도
2. **원본 우체통** (전송 용이성의 핵심): 카톡으로 사진 보내면 화질이 깨지는 게 커플들의 실제 페인포인트인데, "원본 그대로 주고받기"가 그 자리를 대체합니다 — 기록 서비스에 그치지 않는 일상 효용
3. **여행 후 정리 의식**: 하드 연결 → "이번 여행 정리하기" 위저드 (가져오기 → 흐릿 검토 → 대사/설명 쓰기 → Journey 재생으로 마무리) — 게임 감성 컨셉과 결합되는 지점
4. **기록 영속 보장**: 원본이 유실돼도 압축본(2000px)+설명+여정은 영구 — "기록으로서의 제공"이라는 정체성을 명시적 가치로

**구현 우선순위 제안:** 영속화 기반(Supabase + 해시·owner) → ① 보관 현황 → 외장하드 연결+큐 → ② 우체통. ③④는 그 위에 자연히 올라갑니다.

이대로 가면 "원본 관리를 어떻게 하지?"라는 고민이 "장부가 추적하고, 연결되는 순간 맞춰준다"로 정리됩니다. 방향 확정되면 Supabase 스키마부터(이번 논의의 `copies`/`original_status`/우체통 테이블 반영해서) 설계 들어갈게요.

## 1. 맞습니다 — 섞여 있었어요. 정리하면 "두 옵션"이 아니라 "하나의 메커니즘"입니다

제가 혼재되게 설명했는데, 구현 관점에서 정리하면 이렇습니다:

서비스가 만드는 건 **"보관함 연결"이라는 단일 기능 하나**예요 — "폴더 하나를 연결하면, 장부(클라우드 기록)와 그 폴더를 해시로 대조해서 큐(가져오기/삭제/백업)를 실행한다." 그게 전부입니다. **어떤 폴더를 연결하느냐**에 따라 운영 모델이 갈릴 뿐, 코드는 동일합니다:

| 운영 모델 | 연결하는 폴더 | 사람 간 원본 전송 |
|---|---|---|
| **공동 외장하드** (사용자 가정) | 둘 다 같은 외장하드의 `HeartPin/` 폴더 | **하드 자체가 전송 수단** — 우체통 불필요 |
| 각자 관리 | 각자 자기 PC의 사진 폴더 | 우체통(클라우드 임시 중계) 필요 |
| 개인 클라우드 백업 | iCloud Drive/구글드라이브 동기화 폴더 | 위와 동일 + OS가 알아서 클라우드 백업 |

그러니까 "둘 다 제시한다"기보다 — **기능은 하나 만들고, 온보딩에서 '원본을 어디에 모을까요?'를 선택지로 안내**하는 형태가 됩니다. 외장하드로 가정하셨으니 그걸 추천 기본값으로 두고, 우체통(원본 중계)은 **각자 관리 모델을 쓸 때만 필요한 기능이라 후순위로 미루거나 아예 빼도 됩니다.** 공동 외장하드 모델에서는 개발 범위가 그만큼 줄어요.

## 2. 외장하드 연결 — PC 전용 맞고, 이렇게 개발됩니다

별도 네이티브 앱이나 드라이버 없이, **브라우저 표준 API(File System Access API)** 로 웹앱 안에서 됩니다. 외장하드라고 특별한 게 없어요 — OS가 마운트해주면 (`/Volumes/우리하드/` 또는 `E:\`) 브라우저 입장에선 그냥 폴더입니다.

**동작 흐름 (실제 코드 수준):**

```
① 사용자가 [보관함 연결] 클릭
   → showDirectoryPicker({ mode: 'readwrite' })
   → OS 폴더 선택창에서 외장하드의 HeartPin/ 선택 (최초 1회)

② 앱이 DirectoryHandle 획득 — 이 폴더 안에서만:
   - 파일 목록/읽기  → 해시 계산해 장부와 대조 (보유 현황 갱신)
   - 파일 쓰기       → 새 사진 복사, record-backup.json 갱신
   - 파일 삭제       → 흐릿 사진 "버리기" 실행 (removeEntry)

③ 핸들을 IndexedDB에 저장 → 다음 방문부터는
   [다시 연결] 클릭 1회(권한 확인)로 복구 — 폴더 재선택 불필요

④ 하드가 안 꽂혀 있으면 읽기 실패 → "🔌 분리됨" 상태 표시,
   할 일은 큐에 쌓아뒀다가 다시 꽂히면 일괄 실행
```

보안 모델도 깔끔합니다: 사용자가 고른 그 폴더 **밖은 브라우저가 접근 자체를 차단**하고, 삭제 같은 쓰기 권한은 세션마다 사용자 클릭으로 확인받아요.

**지원 범위 (솔직한 제약):**

| 환경 | 보관함 연결 | 비고 |
|---|---|---|
| Chrome/Edge **데스크톱** (맥/윈도우) | ✅ | 유일한 지원 환경 |
| Safari (맥) | ❌ | 폴더 선택 API 미지원 → 맥에서도 크롬 필요 |
| 모바일 (아이폰/안드로이드) | ❌ | API 자체가 없음 |

그래서 개발은 **기능 감지로 분기**합니다: `'showDirectoryPicker' in window`가 참인 환경에서만 "보관함 연결" 버튼을 노출하고, 모바일/사파리에서는 열람·업로드·기록 편집만 제공. 즉 앱이 두 벌이 되는 게 아니라, **같은 웹앱에서 PC 크롬일 때만 정리 기능이 추가로 켜지는** 구조예요.

이 제약이 UX로는 오히려 자연스럽습니다 — 외장하드를 꽂는 행위 자체가 PC 앞이니까, "정리 세션 = PC에서 하드 꽂고 진행"이라는 모델과 정확히 일치합니다.

공수는 이전 추정 그대로: 연결+핸들 영속화+스캔 매칭 ~1일, 큐 실행(복사/삭제/백업) ~0.5~1일. 우체통을 뺀 공동 외장하드 모델 기준으로 전체 그림(Supabase 영속화 + 보관함 연결)이 **~5일 내외**입니다.

## 🤖 Claude

## 1. 모바일 → 외장하드 전송, 서비스가 할 수 있는가? — **됩니다 (클라우드 릴레이 경유)**

폰과 PC가 직접 통신하는 게 아니라, **클라우드가 시간차를 메워주는 우체통**이 됩니다:

```
[폰에서]  사진 선택·업로드
            ├─ 압축본 → 클라우드 (기록용, 영구)
            └─ 원본   → 클라우드 "전송 대기함" (임시)
                              ↓
[나중에 PC에서]  외장하드 꽂고 HeartPin 접속
            → "전송 대기 원본 31장이 있어요" 
            → 일괄 수령: 하드 폴더에 직접 쓰기(FS API)
            → 수령 확인되면 클라우드에서 원본 삭제 ✓
```

즉 폰에서 올리는 순간 기록은 즉시 생기고, 원본은 다음 "하드 연결 세션" 때 자동으로 정본 보관소에 안착합니다. 서비스가 "전송을 맡는다"는 목표 그대로예요.

**단, 모바일 쪽 솔직한 제약 3가지:**
- **데이터 비용**: 원본 200장 ≈ 2~3GB — "원본 전송은 Wi-Fi에서만" 옵션이 사실상 필수
- **백그라운드 업로드 불가**: 모바일 웹은 업로드 중 브라우저를 열어둬야 함 (진행률 UI로 보완)
- **iOS 변수**: 아이폰 사진 선택기가 HEIC→JPEG 변환·메타데이터 처리를 OS 버전/설정에 따라 다르게 해서, **실기기 테스트로 검증이 필요한 영역**입니다 (위치정보가 빠져 들어오면 "위치 없음" 플로우로 자연 수용되긴 합니다)

참고로 이 결정은 이전 스코핑을 하나 뒤집습니다: "공동 외장하드 모델이면 우체통 불필요"라고 했는데, **모바일 업로드를 지원하는 순간 우체통(릴레이)이 폰→하드 수송로로 다시 필요해집니다** (+1~1.5일).

## 2. 외장하드 + 개인 클라우드 + 폰, 동시 사용 가능한가? — **됩니다. 단 역할을 정해야**

기술적으로는 이미 설계한 원장 모델의 자연스러운 일반화입니다. "보관 위치"를 N개로 추적하면 돼요:

```
photo_copies (photo_id, location, status)
locations = 바라폰 · 뇽이폰 · 외장하드 · 바라 iCloud(동기화 폴더) · …
```

다만 **모든 위치를 대등하게 동기화하려 들면 규칙이 폭발**합니다. 권장은 역할을 고정하는 것:

| 위치 | 역할 | 서비스가 하는 일 |
|---|---|---|
| **외장하드** | **정본 보관소 (유일)** | "전체 집합이 여기 모이게 한다" — 빠진 사진을 큐로 수송 |
| 폰 (둘 다) | **출발지(소스)** | 업로드 받기. 하드 안착 후 **"이 사진들은 폰에서 지워도 안전 ✓" 표시** |
| 개인 클라우드 | 개인 사본 (선택) | 직접 연동 X — 동기화 폴더에 써주면 OS가 알아서 올림 |
| 서비스 클라우드 | 기록(압축본) + 우체통 | 영구 보관은 압축본만 |

이러면 동기화 규칙이 단 하나로 줄어듭니다: **"외장하드가 전부를 갖게 하라."** 나머지는 전부 그 규칙의 부산물이에요.

여기서 생기는 보너스 가치 하나 — **"폰 용량 정리"**: 장부가 "어떤 사진이 하드에 안착했는지"를 아니까, "하드에 안전하게 보관된 412장, 폰에서 지워도 돼요"를 보여줄 수 있습니다. 커플들의 실제 페인포인트(폰 용량)를 건드리는 기능이고요. 한계도 명확히: **웹은 폰 카메라롤을 직접 못 지웁니다** — "지워도 안전" 목록을 보여주는 것까지가 서비스의 역할이고, 삭제는 사용자가 사진 앱에서 직접 합니다.

## 정리: 전체 그림과 공수

```
폰(바라·뇽이) ──압축본+원본릴레이──→ 클라우드 ──하드 세션 때 일괄 안착──→ 외장하드(정본)
                                      │                                  ↘ 기록 백업 JSON
                                  압축본=기록(영구)              개인 클라우드 = 동기화 폴더 경유(선택)
```

| 모듈 | 공수 |
|---|---|
| Supabase 영속화 + 해시/owner/copies 추적 | ~3일 |
| 보관함 연결(FS API) + 스캔/쓰기/삭제 큐 | ~1.5일 |
| 우체통(원본 릴레이: 폰→하드 수송) | ~1.5일 |
| 모바일 업로드 다듬기(Wi-Fi 옵션·진행률·iOS 검증) | ~1일 |
| **합계** | **~7일 내외** |

결론: 질문하신 두 가지 모두 "예"이고, 핵심 설계 원칙은 **정본 보관소는 외장하드 하나, 클라우드는 기록+우체통, 폰은 소스** — 이 역할 고정만 지키면 동시 사용이 복잡도 폭발 없이 성립합니다.

"둘이 각자의 기기에서 언제든 기록을 본다"가 요구사항인 이상, 기록(압축본+메타데이터)은 **항상 켜져 있고 양쪽에서 접근 가능한 저장소** = 클라우드에 있어야 합니다. 로컬은 남이 못 보고, 외장하드는 꽂혀 있을 때만 존재하니까요 (하드의 `record-backup.json`은 어디까지나 백업이지 서비스의 동작 기반이 아닙니다).

**하지만 "클라우드여야 한다" ≠ "내(운영자)의 클라우드여야 한다"** — 이게 Q2의 답으로 이어집니다.

## Q2. "틀만 배포, 각자의 클라우드" — 가능하고, 잘 알려진 모델입니다

말씀하신 건 오픈소스 셀프호스팅(BYO-cloud) 모델이고, 지금 아키텍처와 잘 맞습니다. 핵심: **앱이 어느 Supabase를 바라볼지는 설정값(URL + 키) 두 줄**이라서, 그 두 줄만 각 커플의 것으로 갈아끼우면 사진과 기록이 **그 커플의 클라우드 안에서만** 돕니다. 배포 방식 두 가지:

**방식 A — "Deploy 버튼" 템플릿 (완전 분리, 추천)**
```
각 커플이 하는 일 (1회, ~20분):
① Supabase 무료 프로젝트 생성 (서울 리전) → 제공된 SQL 한 번 실행(테이블 생성)
② README의 [Deploy to Vercel] 버튼 클릭 → 자기 계정으로 배포
   (이때 Supabase URL/키를 환경변수로 입력)
③ 끝 — 자기들만의 HeartPin 인스턴스 (앱도, 데이터도, 도메인도 자기 것)
```
- 운영자(당신)는 코드 템플릿만 공개. **사진이 당신 인프라를 1바이트도 거치지 않음**
- 업데이트는 GitHub fork의 "Sync fork" 버튼 → Vercel 자동 재배포

**방식 B — 정적 앱 하나 + 설정 화면**
- 앱은 당신이 배포한 것 하나를 다 같이 쓰되, **첫 실행 마법사에서 각자 자기 Supabase URL/키를 입력** (브라우저에 저장)
- 진입장벽이 더 낮음 (Vercel 계정 불필요, Supabase만 만들면 됨)
- 단, 앱 호스팅은 여전히 당신 소유 — 데이터는 안 거치지만 "앱 제공자" 위치는 유지됨

**당신의 우려에 대한 검증:** 그 직감이 맞습니다. 커플 사진을 운영자 클라우드에 모으는 순간(SaaS 모델) 개인정보처리자로서의 법적 책임·유출 리스크·운영 비용이 전부 당신 몫이 됩니다. BYO 모델은 그걸 구조적으로 제거해요 — "나는 도구를 만들었고, 데이터는 각자의 금고에" 입니다. 비용도 커플당 0원(Supabase·Vercel 무료 티어)이라 과금/정산 문제도 없습니다.

## 지금 설계에 미치는 영향 — 거의 없고, 오히려 지금 정하는 게 이득

이미 하기로 한 설계가 그대로 BYO가 됩니다. 추가로 지켜야 할 원칙만 3가지:

1. **연결 정보를 코드에 안 박고 환경변수/설정으로** (`VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`) — 처음부터 이렇게 짜면 공수 0
2. **스키마를 `supabase/schema.sql` 파일 하나로 유지** — 커플이 SQL 에디터에 붙여넣기 한 번으로 설치
3. **부가 기능(LLM 대사, Mapbox 역지오코딩)은 키 없으면 자동 비활성** — 각자 자기 키를 넣으면 켜지는 옵션 기능으로

우체통(원본 릴레이)도 각 커플의 Supabase Storage를 경유하므로 프라이버시 모델이 일관됩니다. 더 나아가면 **클라이언트 측 암호화(E2EE)** — 업로드 전에 암호화해서 Supabase조차 사진을 못 보게 — 도 가능한데, 키 공유·썸네일 처리 복잡도가 커서 MVP 이후 강화 옵션으로 두는 걸 추천합니다.

**추가 공수:** 첫 실행 셋업 마법사 + 설치 문서(README) **+0.5~1일** 정도. 전체 그림(~7일)에 큰 영향 없습니다.

**정리:** 기록은 클라우드 필수가 맞지만, 그 클라우드를 "각 커플의 것"으로 만드는 게 이 서비스의 프라이버시 정체성과 운영 부담 양쪽의 답입니다. 방식 A(완전 분리)를 기본으로 하되, 당장은 당신 커플의 인스턴스를 첫 번째 배포로 만들면서 위 3원칙만 지키며 개발하면 — 나중에 템플릿 공개는 README 쓰는 일만 남습니다.

## Supabase는 "필수"가 아니라 "공수가 가장 적은 선택"입니다

서비스가 클라우드에 요구하는 건 본질적으로 4가지뿐이에요: ① 구조 데이터 저장(기록 트리), ② 파일 저장(압축본), ③ 두 사람 인증, ④ 브라우저에서 권한 걸고 직접 접근. 이걸 충족하면 뭐든 됩니다:

| 후보 | 비개발자 셋업 난이도 | 비고 |
|---|---|---|
| **Supabase** | 중 — 회원가입+SQL 붙여넣기 (~20분) | 개발 공수 최소. 오픈소스라 **Supabase 자체도 셀프호스팅 가능** = 락인 아님 |
| Firebase | 중 — 비슷한 콘솔 작업 | 동급 대안. 가능하지만 갈아탈 이유는 없음 |
| PocketBase 등 셀프호스팅 | 상 — 서버 직접 운영 | 비개발자에겐 오히려 최악 |
| **개인 클라우드 (Google Drive)** | **하 — 구글 로그인 + 공유폴더 선택, 끝** | 비개발자의 진짜 답. 단 아래 트레이드오프 |
| 운영형 SaaS (당신 호스팅) | 최하 | 프라이버시·운영 부담 때문에 배제하신 옵션 |

## "개발자가 아닌 사람"이 쓴다면 — 솔직한 평가

**Supabase 경로는 셋업 마법사로 아무리 낮춰도 비개발자에겐 부담이 맞습니다** ("Supabase가 뭔데?"에서 막힘). 비개발자까지 간다면 현실적인 답은 **커플이 이미 가진 클라우드 = Google Drive를 백엔드로** 쓰는 겁니다:

```
구글 계정으로 로그인 → 둘이 공유하는 Drive 폴더 선택
   ├─ photos/  압축본 저장
   └─ record.json  기록 트리
```

- 진입장벽: 사실상 0 (이미 있는 계정, 이미 아는 개념인 "공유 폴더")
- 데이터 소유권: 완벽 — 각자 자기 Drive 용량/요금제 안에서, 당신은 여전히 데이터 무접촉
- **대가**: SQL이 없어서 기록을 JSON 파일로 관리(동시 편집 충돌 처리 필요), API 속도 제한, 그리고 **Google OAuth 앱 검증 절차** — 남들에게 배포하려면 구글의 앱 심사(개인정보처리방침, 시연 영상 등)를 통과해야 하는 행정 비용이 있습니다. 개발 공수도 +3~4일

## 추천: 어댑터로 문을 열어두고, 단계적으로

지금 결정할 건 "Supabase냐 Drive냐"가 아니라 **데이터 레이어를 교체 가능하게 짜는 것**입니다:

```
앱 코드 ──→ StorageAdapter 인터페이스 (저장/조회/인증/파일)
              ├─ SupabaseAdapter   ← 1단계: 지금 구현 (우리 커플 + 개발자 템플릿)
              └─ DriveAdapter      ← 2단계: 비개발자 확산을 정말 원할 때
```

- 1단계 공수 영향: **+0.5일** (어차피 하기로 한 데이터 레이어 교체를 인터페이스 규율만 지켜서 하면 됨)
- 여기에 **"통째로 내보내기"(기록 JSON + 사진 zip)** 기능을 넣는 걸 추천합니다 — 어떤 백엔드를 쓰든 "내 데이터는 언제든 들고 나간다"는 신뢰 장치이자, 백엔드 이주의 실질적 보험입니다

**정리하면:** 클라우드는 꼭 Supabase일 필요 없습니다. Supabase는 "개발 공수 최소 + 락인 없음(오픈소스)"이라 1단계로 합리적인 것뿐이고, 비개발자 확산이 목표가 되는 시점의 정답은 Google Drive 어댑터입니다. 지금 할 일은 그 갈아타기가 가능하도록 어댑터 경계 + 내보내기 기능을 처음부터 넣는 것 — 이러면 오늘의 선택이 미래를 안 가둡니다.

여기까지로 아키텍처 그림은 닫힌 것 같아요. 시작하려면 ① Supabase 스키마 + StorageAdapter 인터페이스 설계 → ② 업로드 파이프라인(압축+해시+owner) 순서인데, 진행할까요?

## "모바일 지원"의 실체 — 별도 앱이 아니라 같은 웹앱의 모바일 패스입니다

지금 코드베이스(React 웹) 그대로 **반응형 모바일 웹 + PWA**(홈 화면 설치, 전체화면)로 가면 됩니다. 네이티브 앱을 새로 만드는 게 아니라서, 앞서 논의한 아키텍처(어댑터, 우체통, 보관함 연결)가 전부 그대로 유효해요. 역할 분담도 이미 논의한 대로:

```
모바일 = 열람 + 업로드(카메라롤) + 기록 편집(대사·설명) + "폰에서 지워도 안전" 확인
PC     = 위 전부 + 보관함/외장하드 연결(FS API) + 원본 정리 세션
```

## 공수

| 작업 | 내용 | 공수 |
|---|---|---|
| **모바일 UX 패스** | Main을 "지도 ↔ 기록" 탭/바텀시트로 재배치, 덱 스와이프, 갤러리 터치 넘기기, Journey 모바일 레이아웃(폴라로이드 위치), 안전영역(노치) 처리 | ~2–3일 |
| **PWA화** | manifest + 아이콘 + 서비스워커(앱 셸 캐시) + iOS 설치 안내 | ~0.5–1일 |
| 모바일 업로드 다듬기 | Wi-Fi 옵션·진행률·iOS HEIC/EXIF 실기기 검증 | (기존 ~7일 추정에 이미 포함) |

→ 모바일 추가분 **+3~4일**, 영속화·보관함·우체통 포함 전체 로드맵 **~10일 내외**가 됩니다.

참고: 프로토타입에 기본 반응형(좁은 화면 세로 스택)은 있지만, "모바일에서 쓸 만한" 수준은 위 UX 패스가 필요합니다 — 특히 지도+기록 패널을 세로로 쌓으면 둘 다 좁아져서, 모바일은 탭 전환이나 지도 위 바텀시트가 맞아요.

## 클라우드 호환 — 둘 다 모바일에서 그대로 됩니다

어댑터 경계는 UI와 **직교**라서 모바일 여부와 무관합니다:

| | 모바일 브라우저에서 | 주의점 |
|---|---|---|
| **Supabase** | ✅ SDK가 그냥 HTTPS 호출 — PC와 완전 동일 | 없음 |
| **Google Drive** | ✅ OAuth + REST 모두 동작 | 로그인은 팝업 대신 **리다이렉트 방식**으로 (iOS 팝업 차단 회피) |

모바일에서 달라지는 건 클라우드가 아니라 **브라우저 능력**입니다:

- **FS Access API 없음** → 보관함/외장하드 연결·로컬 삭제는 PC 전용 (이미 확정한 분담)
- **백그라운드 업로드 없음** → 업로드 중 화면 유지 필요 (진행률 UI로 보완)
- **iOS 저장소 퇴거 정책**: 사파리는 7일간 미방문한 사이트의 localStorage를 지울 수 있음 → 로그인 세션이 풀리는 정도(데이터는 클라우드라 무손실)지만, **PWA로 홈 화면에 설치하면 이 제약이 사실상 사라집니다** — PWA화를 권장하는 실질적 이유

## 네이티브 앱은 언제?

"폰 카메라롤 자동 감지·동기화"나 "폰에서 사진 삭제까지 서비스가 직접"이 꼭 필요해지는 시점에만 Capacitor(웹앱을 그대로 감싸는 방식)로 — 그때도 코드 대부분 재사용되지만 앱스토어 심사·연 $99 계정 등 운영 부담이 생기고, "틀 배포" 모델과도 결이 안 맞아서 **지금은 비추천**입니다. 모바일 웹+PWA로 충분히 커버돼요.

**정리:** 모바일 지원은 +3~4일짜리 같은 웹앱의 확장이고, Supabase든 Drive든 어댑터 그대로 모바일에서 동작합니다. 아키텍처 결정 중 모바일 때문에 바뀌는 것은 없습니다 — 이제 정말 그림이 닫혔으니, 구현 들어갈 준비 되면 ① Supabase 스키마 + StorageAdapter 인터페이스부터 시작하겠습니다.

## 스토어 앱으로 가는 법 — 기존 코드를 거의 다 재사용합니다

**Capacitor** 방식(추천): 지금 만든 React 웹앱을 네이티브 껍데기로 감싸서 그대로 앱 binary로 만듭니다. 코드 재사용률 ~90% 이상이라, **웹으로 먼저 개발해도 버려지는 게 없어요.** (반대로 React Native로 다시 짜는 건 +1~2달이라 비추천)

추가 공수 내역 (iOS+Android 합산, 대략 **+1.5~2주**):
- Xcode/Android Studio 빌드 환경 + Capacitor 셋업: ~1일
- 네이티브 연동(카메라롤 플러그인, 푸시 등): ~3~5일
- 아이콘·스플래시·스토어 등록 자료: ~1일
- 심사 대응·출시 행정: Apple 개발자 계정 **연 $99** + Google **$25**, 심사 대기, **업데이트할 때마다 재심사**

## 스토어 앱이어야만 얻는 것 — 사실 이게 핵심 판단 기준입니다

1. **카메라롤 깊은 통합**: 웹은 사진 "선택"만 되지만, 앱은 권한 받고 전체 카메라롤을 읽을 수 있어 **EXIF(날짜·GPS)가 확실하게 보존**되고(웹의 iOS 변수 제거), 무엇보다 — **폰에서 흐릿한 사진 삭제가 가능해집니다.** 웹에서는 불가능해서 "지워도 안전 목록만 보여주기"로 후퇴했던 그 기능이요. 정리 서비스라는 정체성에 이건 꽤 큰 차이입니다.
2. **푸시 알림**: "뇽이가 사진 12장을 올렸어요", "우체통에 원본 도착" — 커플 서비스의 생명력
3. **배포 친숙함**: 비개발자에게 "스토어에서 받아" 한마디면 끝

반대로 **솔직한 리스크**: Apple은 "웹사이트를 그냥 감싼 앱"을 리젝하는 경우가 있어서(최소 기능성 조항), 위 1·2 같은 네이티브 기능을 넣은 상태로 심사받는 게 안전합니다. 그리고 스토어에 올리는 순간 당신이 "퍼블리셔"가 됩니다 — 데이터는 여전히 각 커플 클라우드라 무접촉이지만, 개인정보처리방침 게시·심사 대응 같은 행정은 생겨요. (BYO 모델과는 양립합니다: **앱은 하나 출시하고, 각 커플이 첫 실행에서 자기 Supabase/Drive를 연결** — Capacitor 안도 같은 JS 코드라 어댑터가 그대로 동작합니다.)

## 추천 로드맵

```
1단계  웹/PWA로 전 기능 완성·검증  (영속화+보관함+우체통+모바일 UX ≈ 10일)
2단계  Capacitor로 래핑 → 카메라롤·푸시 추가 → 스토어 출시  (+1.5~2주)
```

이 순서를 추천하는 이유: ① 같은 코드라 1단계가 전혀 낭비가 아니고, ② 스토어 심사라는 느린 루프에 들어가기 전에 빠른 웹 배포로 기능을 다듬을 수 있고, ③ 2단계 시점엔 "앱이어야만 하는 기능"(폰 사진 삭제·푸시)만 얹으면 되어 심사 리스크도 줄어듭니다.

**직답하면:** 스토어 다운로드 앱, 됩니다. 단 "웹앱 대신"이 아니라 "웹앱을 완성한 뒤 그걸 감싸서"가 가장 싸고 안전한 경로이고, 추가 공수는 +1.5~2주 + 연간 운영(계정비·심사)입니다.

## 어떻게 가능한가 — "반응형"을 넘어선 "적응형"

두 가지 기법을 섞습니다:

```
① CSS 미디어쿼리          — 같은 컴포넌트가 크기만 재배치되는 부분 (카드 그리드 등)
② 조건부 렌더링            — 구조 자체가 다른 부분:
   isMobile
     ? <BottomTabBar /> + <MapWithBottomSheet />   ← 모바일 전용 셸
     : <Topbar /> + <Map + RNB 패널 />              ← 현재 데스크톱 셸
```

핵심은 **셸(뼈대)만 두 벌이고 내용물은 공유**된다는 점입니다. 말풍선·사진 카드·덱·정리함 카드·Journey·데이터 레이어는 전부 그대로 쓰고, "어디에 어떻게 배치하느냐"만 갈라져요. 그래서 앱 두 개를 만드는 공수가 아니라 +2~3일인 겁니다.

배포도 하나입니다 — 같은 URL을 폰으로 열면 하단 탭바 UI, PC로 열면 지금 UI가 자동으로 나옵니다.

## HeartPin 모바일 셸 스케치

```
┌─────────────────┐
│                 │   지도 = 전체 화면
│      지도        │   기록 = 아래에서 끌어올리는 바텀시트
│                 │     (카드 덱·타임라인이 시트 안으로)
│  ╭───────────╮  │
│  │ ⌃ 해운대…  │  │ ← 바텀시트 (드래그)
├──┴───────────┴──┤
│ 🏠   🗺️   ＋   🗂️ │ ← 하단 내비: 홈 · 지도 · 업로드 · 정리함
└─────────────────┘
```

- **하단 탭바**: 홈 / 지도(여정) / ＋업로드 / 정리함 — 데스크톱의 상단바·RNB 진입점들이 여기로 이동
- **Main**: RNB 사이드패널 → **지도 위 바텀시트**로 (모바일 지도앱의 표준 패턴)
- **Journey·갤러리**: 이미 풀스크린 구조라 위치 조정 정도
- **정리함·홈**: 그리드가 1열로 — 이건 ①번 미디어쿼리로 충분

## 그리고 이게 2단계(스토어 앱)에 그대로 들어갑니다

Capacitor는 이 웹앱을 그대로 감싸므로, **지금 만드는 모바일 UI = 나중에 스토어 앱에서 보이는 UI**입니다. 모바일 UX를 지금 만들어두면 2단계에서 화면 작업은 사실상 0이고, 네이티브 기능(카메라롤·푸시)만 얹으면 돼요.

**직답:** 네, 한꺼번에 개발 가능합니다. "PC 화면을 욱여넣은 모바일"이 아니라 하단 내비·바텀시트를 갖춘 진짜 모바일 앱 모양으로요. 다만 셸 두 벌 + 바텀시트 인터랙션까지 제대로 하면 공수는 앞서 말한 범위의 윗단(**~3일, 디테일 욕심내면 +1일**)으로 보시면 됩니다.

## 문서 구성 (7개 섹션)

1. **서비스 정의와 가치** — 맡는 것(기록·전송·소재추적) / 맡지 않는 것(원본 백업·SaaS 운영) + **원장(Ledger) 모델** 개념
2. **아키텍처 결정 D1~D11** — 각 결정 + 근거 + 배제한 대안 (ADR 형식: "왜 FastAPI 아닌가", "왜 BYO-cloud인가" 등 논의 경로 보존)
3. **시스템 구조** — 데이터 흐름도(폰→클라우드 기록/우체통→외장하드), 위치별 역할 고정표, Supabase 스키마 초안(`content_hash`·`owner`는 Phase 1부터 필수 표시), StorageAdapter 경계
4. **로드맵 5단계**:
   - **Phase 1 (MVP, ~5–6일)**: 말씀하신 정의 그대로 인용 — 외장하드 연결 + 웹 업로드 기록 + 상대방 접속·업로드. 작업 5개 + 완료 기준(DoD) 체크리스트 + **의도적으로 뺀 것** 명시
   - Phase 2 모바일(~4–5일) / Phase 3 전송·공동관리(~3–4일) / Phase 4 스토어 앱(+1.5–2주) / Phase 5 개방·확장(수시)
5. **제약과 리스크** — FS API 브라우저 제약, iOS 변수, Supabase 1GB 등 + 대응
6. **미결 사항** — 압축 파라미터, E2EE 키 공유 방식 등 추후 논의 7건
7. **논의 기록 요약** — 오늘 대화가 결정에 이른 경로 10단계 (추후 "왜 이렇게 했지?"의 답)

## 채팅 내용 내보내기 — 가능합니다

Claude Code 내장 명령인 `/export`를 사용하면 됩니다 (직접 입력해 주셔야 해요):

```
/export docs/CHAT-2026-06-05.md
```

이렇게 파일명을 주면 대화 전체가 레포 안 `docs/` 폴더에 마크다운으로 저장됩니다. (그냥 `/export`만 치면 클립보드 복사도 선택 가능)

**다른 PC에서 작업하실 거라면 추천 흐름:**

1. 지금 `/export docs/CHAT-2026-06-05.md` 실행 → 저장되면 말씀해 주세요, 제가 커밋·푸시할게요 (직접 하시려면 `git add docs/ && git commit -m "docs: 설계 논의 채팅 로그" && git push`)
2. 다른 PC에서:
   ```bash
   gh auth login          # wisroo 계정으로
   git clone https://github.com/wisroo/HeartPin.git
   cd HeartPin && npm install
   ```
3. 그쪽에서 Claude Code를 열고 "docs/ROADMAP.md와 docs/CHAT-2026-06-05.md를 읽고 Phase 1부터 시작하자"라고 하면 오늘 맥락 그대로 이어집니다

**참고 두 가지:**
- 세션 자체(`claude --resume`)는 이 맥에 로컬이라 다른 PC로 이어지지 않습니다 — 그래서 채팅 로그를 레포에 넣는 위 방식이 정답이에요
- 사실 핵심 결정사항은 이미 `ROADMAP.md`에 다 정리돼 있어서, 채팅 로그는 "논의의 세부 뉘앙스"가 필요할 때를 위한 보험입니다. 레포는 private이라 올려도 안전합니다
